## The problem

A hackathon platform coordinates more than registrations. Teams form, submissions move through deadlines, evaluators score work, moderators handle exceptions, and scheduled events trigger notifications. Putting these concerns directly into controllers would make every lifecycle change expensive.

## Architecture

The system applies Clean Architecture so domain rules and use cases remain independent from Spring Boot, H2, and delivery concerns. Domain events communicate meaningful changes without forcing unrelated modules to depend directly on each other.

## Role and contribution

Developed as a Software Engineering course project by Matteo Vittori and Alejandro Innocenzi, HackHub covers backend design and development, Clean Architecture, SOLID principles, event-driven scheduling and notifications, a test-oriented frontend, and UML modelling in Visual Paradigm.

## Important decisions

- Model lifecycle transitions as explicit use cases rather than controller scripts.
- Use domain listeners for cross-cutting reactions such as notifications.
- Keep persistence models outside the domain boundary.
- Use a scheduler for deadline-driven changes instead of relying on user traffic.
- Maintain visual use-case, class, and sequence models alongside implementation.
- Use Strategy adapters for calendar and payment integrations.
- Model support requests and user accounts with explicit State transitions.
- Inject a centralised clock so lifecycle and scheduler behaviour remain deterministic in tests.

## Delivery and API

The Java 21 and Spring Boot 3 application exposes OpenAPI documentation locally through Swagger UI and uses a file-backed H2 database for its development setup. A separate testing frontend connects to the local REST API and exercises the main workflows. The source is published under the MIT License.

## Challenges and trade-offs

Layered boundaries add mapping and interface overhead to a course-sized application. That cost becomes worthwhile when business rules, persistence, and delivery mechanisms are expected to change independently.

## Results and lessons

The project demonstrates how Clean Architecture changes the location of decisions: Spring Boot becomes an adapter, use cases own orchestration, domain objects carry invariants, and external services remain replaceable strategies.
