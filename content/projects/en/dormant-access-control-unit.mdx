## The problem

An access controller should react immediately when needed without keeping every peripheral active continuously. It must coordinate proximity sensing, authentication, display feedback, door control, failed attempts, and alarms while remaining predictable on resource-constrained hardware.

This prototype keeps the controller dormant until the Watchdog Timer requests a proximity measurement. Detection within 30 centimetres wakes the interaction flow and presents a Morse-code challenge through a single push button.

## Operating flow

1. The unit remains in low-power sleep mode.
2. The Watchdog Timer periodically activates the HC-SR04 sensor.
3. A nearby user wakes the system and the display shows `CIAO` (hello).
4. Short and long button presses are translated into Morse dots and dashes.
5. A valid code displays `Ent` and activates the door relay.
6. Invalid attempts display `Er n`; repeated failures flash `8888` and activate the buzzer.
7. Completion or timeout returns the controller to sleep.

## Modular architecture

The implementation separates the ultrasonic sensor, seven-segment display, Morse input handler, access-control rules, alarm management, and sleep manager. Each module owns one hardware or behavioural responsibility, reducing coupling between timing logic and domain decisions.

Interrupts replace blocking waits throughout the flow:

* Watchdog Timer schedules proximity checks.
* `INT0` captures the ultrasonic echo for distance measurement.
* `INT1` captures Morse-code button input.
* `Timer0` generates the trigger signal.
* `Timer1` multiplexes the four-digit display.
* `Timer2` manages press durations and interaction timeouts.

## Timing model

* Button debounce: 10–50 ms.
* Morse dot: 100–300 ms.
* Morse dash: 300–900 ms.
* Input timeout: approximately 6 seconds.
* Display refresh: at least 40 Hz.
* Alarm buzzer: approximately 750 Hz.

## Trade-offs

The HC-SR04 provides explicit distance thresholds but requires more active measurement than a PIR sensor. A four-digit display communicates system state more clearly than LEDs, while adding multiplexing and timer complexity. Morse input keeps the hardware minimal, but requires careful timing boundaries and more user learning than a keypad.

## Outcome

Two Wokwi configurations expose the normal interaction and a debug-oriented mode. The project demonstrates modular embedded design, hardware-software coordination, interrupt management, and low-power control logic. Its efficiency claim is architectural rather than a measured battery-life result; physical power profiling would be the next validation step.
