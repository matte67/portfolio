## The problem

Video-analysis experiments often begin as focused scripts and gradually absorb acquisition, tracking, metrics, visualisation, configuration, and output logic. That works for one experiment; it becomes expensive when an algorithm changes or a result must be reproduced.

SEF treats this as an architectural problem. Its goal is not to replace specialised computer-vision libraries, but to give them explicit boundaries and a shared execution model.

<PipelineExplorer />

## A stable core, replaceable edges

The core owns contracts, typed artifacts, runtime planning, events, plugin registration, errors, and outputs. OpenCV, Matplotlib, YOLO, ArUco, Streamlit, and other concrete tools remain adapters. Dependencies point inward so presentation and infrastructure can evolve without rewriting the rules at the centre.

Matteo led the framework architecture and implemented the core, runtime, CLI, user interfaces, and documentation. Alejandro focused primarily on the concrete computer-vision plugins. This division made the central contracts and their adapter implementations evolve as two explicit parts of the same system.

API, CLI, and SEF Studio expose the same internal concepts. A pipeline can be composed in Python, described declaratively in YAML or JSON, or configured visually without duplicating orchestration logic.

## Runtime decisions

Each stage declares capabilities. The planner can preserve a batch workflow, produce progressive streaming results, or materialise only where a stage requires the complete sequence. This keeps one public model while allowing different memory and latency profiles.

### Important trade-offs

- Explicit contracts introduce a small abstraction cost, but isolate algorithms and make replacement safer.
- Streaming reduces memory pressure, while some global algorithms still require materialisation.
- A registry improves extension and declarative configuration, but naming and versioning become part of the public design.
- UI-agnostic outputs increase reuse, while the presentation layer must interpret richer artifact metadata.

## Results and lessons

The reference benchmark reported streaming execution at 1.01× the direct loop while reducing peak memory from 154.7 MiB to 39.8 MiB. More importantly, the same architecture supported batch, streaming, configurable components, progressive events, and multiple presentation layers.

The published benchmark suite uses synthetic NumPy frames, ten measured repetitions after warm-up, and the median result on a MacBook with an Apple M2 processor and 8 GB of memory. Raw data and reproducible benchmark commands are kept with the source repository.

SEF currently targets Python 3.11 or newer and has been tested on macOS and Windows. It remains experimental and pre-1.0 while its public API is hardened for a stable release. The project is seeking real adopters, particularly around ENEA’s motion-magnification use case for monitoring structures and historic monuments.

## Declarative pipeline example

The same runtime can be configured without embedding orchestration in application code. This example extracts dense optical flow and generates a vector field and magnitude heatmap.

```json
{
  "schema_version": "1.0",
  "pipeline": {
    "frame_extractor": {
      "name": "opencv_buffered",
      "params": {
        "path": "/your/path/to/video.mp4",
        "config": { "max_frames": null, "stride": 1 }
      }
    },
    "frame_processors": [],
    "signal_extractor": {
      "name": "opencv_dense_optical_flow",
      "params": { "cell_size": 16 }
    },
    "signal_cleaners": [],
    "analyzers": [
      {
        "name": "dense_optical_flow_vector_field",
        "params": { "config": { "sample_index": -1, "normalize": false } }
      }
    ],
    "visualizers": [
      {
        "name": "optical_flow_vector_field",
        "params": { "config": { "color_by_magnitude": true } },
        "result_indices": [0]
      },
      {
        "name": "optical_flow_heatmap",
        "params": { "config": { "mode": "magnitude", "cmap": "viridis" } },
        "result_indices": [0]
      }
    ]
  }
}
```

The strongest lesson is that the useful abstraction is not a new vision algorithm. It is the explicit infrastructure that lets different algorithms coexist, remain inspectable, and evolve independently.
