## Il problema

Gli esperimenti di analisi video spesso nascono come script circoscritti e finiscono per inglobare acquisizione, tracking, metriche, visualizzazione, configurazione e logica di output. Questo approccio può bastare per un singolo esperimento, ma diventa costoso quando cambia un algoritmo o bisogna riprodurre un risultato.

SEF affronta il problema dal punto di vista architetturale. Non vuole sostituire le librerie specializzate di computer vision, ma offrire loro confini espliciti e un modello di esecuzione condiviso.

## Un core stabile, estremità sostituibili

Il core definisce contratti, artefatti tipizzati, pianificazione del runtime, eventi, registrazione dei plugin, errori e output. OpenCV, Matplotlib, YOLO, ArUco, Streamlit e gli altri strumenti concreti restano adapter. Le dipendenze puntano verso l’interno, così presentation e infrastructure possono evolvere senza riscrivere le regole centrali.

Matteo ha guidato l’architettura del framework e implementato core, runtime, CLI, interfacce utente e documentazione. Alejandro si è concentrato soprattutto sui plugin concreti di computer vision. Questa divisione ha permesso di far evolvere i contratti centrali e le rispettive implementazioni come due parti esplicite dello stesso sistema.

## Un modello, tre punti di accesso

API, CLI e SEF Studio espongono gli stessi concetti interni. Una pipeline può essere composta in Python, descritta in modo dichiarativo con YAML o JSON oppure configurata visivamente, senza duplicare la logica di orchestrazione.

## Decisioni di runtime

Ogni fase dichiara le proprie capacità. Il planner può preservare un flusso batch, produrre risultati progressivi in streaming oppure materializzare i dati soltanto quando una fase richiede la sequenza completa. Il modello pubblico resta unico, pur supportando profili diversi di memoria e latenza.

### Compromessi principali

- I contratti espliciti introducono un piccolo costo di astrazione, ma isolano gli algoritmi e ne rendono più sicura la sostituzione.
- Lo streaming riduce la pressione sulla memoria, mentre alcuni algoritmi globali richiedono comunque la materializzazione.
- Un registry semplifica estensione e configurazione dichiarativa, ma nomi e versioni diventano parte dell’interfaccia pubblica.
- Output indipendenti dalla UI aumentano il riuso, richiedendo alla presentation di interpretare metadati più ricchi.

## Risultati e lezioni apprese

Nel benchmark di riferimento, l’esecuzione in streaming ha richiesto 1,01 volte il tempo del ciclo diretto, riducendo al contempo il picco di memoria da 154,7 MiB a 39,8 MiB. Il risultato più importante è però architetturale: lo stesso modello supporta batch, streaming, componenti configurabili, eventi progressivi e più livelli di presentazione.

La suite pubblicata usa frame NumPy sintetici, dieci ripetizioni misurate dopo il warm-up e il valore mediano su un MacBook con processore Apple M2 e 8 GB di memoria. Dati grezzi e comandi riproducibili sono conservati nel repository.

SEF richiede attualmente Python 3.11 o successivo ed è stato provato su macOS e Windows. Rimane un progetto sperimentale pre-1.0, mentre l’API pubblica viene consolidata in vista di una release stabile. Il progetto cerca primi utilizzatori reali, in particolare nell’ambito del motion magnification per il monitoraggio di strutture ed edifici storici collegato a ENEA.

## Esempio di pipeline dichiarativa

Lo stesso runtime può essere configurato senza incorporare l’orchestrazione nel codice applicativo. Questo esempio estrae l’optical flow denso e genera un campo vettoriale e una heatmap dell’intensità.

```json
{
  "schema_version": "1.0",
  "pipeline": {
    "frame_extractor": {
      "name": "opencv_buffered",
      "params": {
        "path": "/your/path/to/video.mp4",
        "config": { "max_frames": null, "stride": 1 }
      }
    },
    "frame_processors": [],
    "signal_extractor": {
      "name": "opencv_dense_optical_flow",
      "params": { "cell_size": 16 }
    },
    "signal_cleaners": [],
    "analyzers": [
      {
        "name": "dense_optical_flow_vector_field",
        "params": { "config": { "sample_index": -1, "normalize": false } }
      }
    ],
    "visualizers": [
      {
        "name": "optical_flow_vector_field",
        "params": { "config": { "color_by_magnitude": true } },
        "result_indices": [0]
      },
      {
        "name": "optical_flow_heatmap",
        "params": { "config": { "mode": "magnitude", "cmap": "viridis" } },
        "result_indices": [0]
      }
    ]
  }
}
```

La lezione principale è che l’astrazione utile non consiste in un nuovo algoritmo di visione. È l’infrastruttura esplicita che permette ad algoritmi diversi di coesistere, restare analizzabili ed evolvere in modo indipendente.
